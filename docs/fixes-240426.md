# Findings & Open Questions — 2024-04-26

This document records the bugs fixed, the edge-case behaviours discovered (some intentional, some potentially surprising), and a few open questions worth discussing before the next commit.

---

## Fixed bugs (already in C code, committed)

### 1. `rename_branch` to existing name was silently corrupt
**Before**: Renaming branch A to a name that already existed would succeed without error, leaving two entries with the same name in the branch table. Subsequent `pragma branches` would show duplicates.  
**After**: Returns `SQLITE_ERROR` ("unable to rename branch") if the target name already exists on a *different* branch.  
**Test**: `test_errors.py::test07_rename_to_existing_name_errors`

### 2. `rename_branch` identity rename was broken by the fix above
**Before (our fix)**: The duplicate-name check `if sqlite3BranchFind(lmdb, zNewName) > 0` also rejected renaming a branch to its own name (the branch itself is found by name, so it "exists").  
**After**: Check is `existing_id != id`, so renaming master→master is a silent no-op.  
**Test**: `test_branch_ops.py::test11_rename_identity_is_noop`, `test_advanced.py::test04_rename_identity_is_noop`

### 3. `branch_truncate` beyond head was a silent no-op
**Before**: `if to_commit >= branch->last_commit return SQLITE_OK` — any commit number at or above head silently did nothing, so truncating to commit 999 on a 3-commit branch succeeded without error and left data untouched.  
**After**: `if to_commit > branch->last_commit return SQLITE_ERROR`.  
**Test**: `test_errors.py::test13_truncate_beyond_head_errors`

## Documented behaviours (not bugs, but worth knowing)

### 4. Write-to-historical-commit error surfaces at `conn.commit()`, not `c.execute()`
SQLite defers the "access permission denied" until flush time, so:
```python
c.execute("pragma branch=master.1")
c.execute("insert into t values ('x')")   # no error here
conn.commit()                             # OperationalError raised here
```
This is a SQLite pager design constraint, not a LiteTree bug.  
**Status**: Documented in README. Tests reflect this (`test_errors.py::test04`).

### 5. Branch names starting with digits are allowed
The implementation has no restriction on the first character (beyond the overall name length and dot/space exclusions). `pragma new_branch=1branch` succeeds.  
**Status**: Documented in `test_branch_ops.py::test07`. **Open question**: should we add validation to reject digit-leading names for consistency with common identifier rules?

### 6. Deleting a branch used by another connection raises `InternalError`, not `OperationalError`
When connection B is on branch X and connection A deletes branch X, the next query on connection B raises `sqlite3.InternalError: unknown operation` (not `OperationalError`).  
**Status**: Test catches both. **Open question**: should this be `OperationalError` for a more user-friendly API? Requires C-side change to map the internal error code.

### 7. `new_branch` without `at` uses branch HEAD, not current historical commit
```python
c.execute("pragma branch=mid.2")       # viewing historical commit
c.execute("pragma new_branch=foo")     # source_commit = mid HEAD (3), not 2
```
To branch from a historical point you must use the explicit `at` form:
```python
c.execute("pragma new_branch=foo at mid.2")
```
**Status**: Documented in test. **Open question**: arguably `new_branch` without `at` while on a historical commit should use that historical commit as source. Currently it ignores it. Worth discussing.

### 8. Deleting a parent branch leaves `source_branch` in child `branch_info` as a stale/dangling string
After `del_branch(master)`, child branches still work (they reference the parent by integer ID internally), but `branch_info(child)["source_branch"]` returns the old string "master" even though that branch is gone.  
**Status**: Documented in `test_branch_ops.py::test15`. **Open question**: should `branch_info` detect and report stale parent references?

### 9. Branch name hard limit is 127 characters (not 128)
`char name[128]` in the C struct means 127 usable characters + null terminator. A 127-char name succeeds; 128-char raises `OperationalError`.  
**Status**: Documented in tests and CLAUDE.md.
